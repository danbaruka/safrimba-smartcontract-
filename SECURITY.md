# Security policy

## Supported versions

We release patches for security vulnerabilities for the current stable release and the immediately prior minor release when feasible.

| Version | Supported          |
| ------- | ------------------ |
| 0.1.x   | :white_check_mark: |
| < 0.1   | :x:                |

## Reporting a vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

If you believe you have found a security issue in the Safrimba smart contract or related tooling:

1. **Report privately** by opening a [GitHub Security Advisory](https://github.com/your-org/safrimba-smartcontract/security/advisories/new) (preferred), or by emailing the maintainers if you have a contact address.
2. Include a clear description, steps to reproduce, and impact.
3. Allow a reasonable time for a fix before any public disclosure.

We will acknowledge your report as soon as possible and will keep you updated on the status of the fix. We appreciate responsible disclosure and will credit reporters (unless they prefer to remain anonymous) when the issue is disclosed.

**Do not open public issues for unfixed security vulnerabilities.** Public discussion can put users at risk.

## Security update process

- Critical issues will be addressed as a priority; we aim to provide an initial response within a few days.
- Fixes may be developed in a private fork and released in a patch version before public disclosure.
- After a fix is released, we may publish an advisory and update the changelog.
